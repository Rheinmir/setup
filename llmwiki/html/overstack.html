overstack

Lớp khung biến AI Agent thành một cộng sự kỹ thuật tự-kỷ-luật: tri thức (wiki) · rào chắn tất định (harness, 14 rule, 0 token) · kỹ năng (65 skill) · điều phối đa-agent (Orca). Bắt đầu ở Quickstart; các tab sau đi sâu từng phần.

00 · Bắt đầu

Quickstart — chạy được trong 2 phút

Bạn chỉ cần nhớ ba thứ: cài, để agent làm việc, và khi muốn tính năng mới thì /propose trước. Phần còn lại overstack lo (rào chắn tất định chặn agent làm bậy, 0 token).

1. Cài vào dự án của bạn (một dòng)

Cách 1 — dán cho Agent (agent tự cài rồi tự kiểm tra mọi thứ đã đúng chỗ):

chạy curl -fsSL https://raw.githubusercontent.com/Rheinmir/setup/orca/harness/poc-vendor-neutral/bootstrap.sh | bash và kiểm tra xem mọi thứ đã ở đúng chỗ chưa

Cách 2 — chạy thẳng trong terminal:

curl -fsSL https://raw.githubusercontent.com/Rheinmir/setup/orca/harness/poc-vendor-neutral/bootstrap.sh | bash

Lệnh này kéo cả ba trụ — harness (rào chắn), skills (kỹ năng), llmwiki (nền tri thức) — và bật guardrail ngay. Dự án cũ đã có wiki? Gọi /harness-update để migrate + tự trả nợ.

2. Cứ làm việc bình thường với agent

Agent (Claude Code, Cursor, opencode…) giờ bị overstack gác: không ghi bậy vào raw/, mọi trang wiki buộc có nguồn gốc (## Origin), index/log tự cập-nhật-bằng-code. Bạn không phải nhớ luật — hàng rào tự cắn.

3. Muốn một tính năng/đổi thay? gọi /orca-workflow

Đây là lệnh bạn dùng nhiều nhất. /orca-workflow tự chạy trọn vòng propose → gate → dispatch → verify: vẽ kế hoạch (file ảnh hưởng, cái gì có thể vỡ, sequence diagram) + DỪNG chờ bạn duyệt, rồi giao việc cho các agent chạy song song, rồi đối chiếu kết quả. Các skill con (propose, safe-change, verify-before-commit…) do nó điều phối — bạn hiếm khi gọi tay từng cái.

Lệnh bạn gọi TRỰC TIẾP nhiều nhất

  • /orca-workflow — dựng/đổi một tính năng (điều phối cả propose→gate→dispatch→verify).
  • /orca-onboard — onboard codebase (một hay nhiều) song song vào wiki.
  • /harness-update — tự bảo trì: cập nhật overstack + trả nợ wiki + health-check.
  • find-skills "<việc>" — không chắc có sẵn kỹ năng gì? hỏi nó.

Cần hiểu sâu từng phần? Các tab bên trái đi từ tổng quan → cài đặt → ba trụ → workflow → tự bảo trì → tham chiếu đầy đủ.

01 · Tổng quan

overstack là gì

overstack không phải một app — nó là một lớp khung (a stack you put over your project) biến AI Agent thành một cộng sự kỹ thuật tự-kỷ-luật: có trí nhớ (wiki), có nguyên tắc không thể phá (harness), có tay nghề đóng gói sẵn (skills), và biết điều phối nhiều agent (Orca).

Bốn trụ

  • Wiki (tri thức) — nơi dự án nhớ: concept, entity, nguồn, ADR, draft; mỗi trang truy được nguồn.
  • Harness (rào chắn) — luật tất định bằng code chặn agent làm bậy, không tốn token, không bypass được khi merge.
  • Skills (kỹ năng) — quy trình đóng gói, gọi bằng /tên, cài global dùng mọi dự án.
  • Orca (điều phối) — chạy nhiều agent song song: propose → gate → dispatch → verify.

Triết lý

  • Đừng tin agent nhớ — luật, log, bản đồ năng lực đều do CODE đảm bảo, không trông chờ trí nhớ LLM.
  • Tất định > xác suất — rào chắn là code chạy 0-token; phần cần LLM bị nhốt sau adapter.
  • Build-now-adapt-later — dựng phần chắc chắn ngay, nhốt ẩn số sau MỘT adapter để chỉnh sau.
  • Travel-with-project — cái gì phục vụ dự án thì đi theo khi cài; đồ nghề dev framework ở lại.
02 · Cài đặt

Cài đặt — global vs per-project

overstack cài bằng một dòng bootstrap. Có hai chế độ, dùng cả hai là tốt nhất.

Bootstrap (mặc định — cả 3 trụ)

Cách 1 — dán cho Agent (agent tự cài + tự kiểm tra):

chạy curl -fsSL https://raw.githubusercontent.com/Rheinmir/setup/orca/harness/poc-vendor-neutral/bootstrap.sh | bash và kiểm tra xem mọi thứ đã ở đúng chỗ chưa

Cách 2 — terminal:

curl -fsSL https://raw.githubusercontent.com/Rheinmir/setup/orca/harness/poc-vendor-neutral/bootstrap.sh | bash

Kéo harness + skills + llmwiki và bật chặn ngay. Opt-out: --harness-only.

Dự án MỚI từ đầu (chưa có code)?

Không cần cài tay rồi feed từng file — chỉ dán nội dung 00-New-Project.md cho agent. Nó tự cài overstack → kickoff (hỏi 3 câu) → dựng knowledge base → scaffold MVP, dừng hỏi đúng lúc. Chi tiết: setup.md.

Per-project (cho team)

  • harness/ + .claude/settings.json được commit vào repo → đồng đội clone về là được bảo vệ.
  • Có pre-commit (L2) + baseline audit cho dự án cũ.
  • Đây là cách DUY NHẤT bảo vệ cả team (global chỉ bảo vệ máy bạn).

Global (cho máy bạn)

  • install-harness.sh --global → hooks + validators vào ~/.claude/harness/.
  • Mọi dự án có llmwiki/ trên máy được gác; dự án khác chỉ tốn ~1ms/tool-call.
  • Kèm code-logger + build-capabilities (ADR-005) → log + bản đồ năng lực xuống cùng dự án.

Sau khi cài

  • Mở session mới thì hooks mới nạp.
  • Chưa có pre-commit? cài pipx install pre-commit để bật chặn ở lớp L2.
  • Tự bảo trì về sau: /harness-update (cập nhật + trả nợ + refresh bản đồ năng lực).
03 · Tri thức

Trụ 1 — Wiki (nền tri thức)

Wiki là bộ nhớ dài hạn của dự án. overstack ép nó luôn truy được nguồn và không bao giờ lệch index — bằng rào chắn, không bằng kỷ luật con người.

Luật vàng của wiki

  • Mọi trang phải có ## Origin — luôn truy được nguồn gốc (R2).
  • Thêm/xoá trang phải cập nhật wiki/index.md (R3) — overstack có --fix tự thêm.
  • Mỗi thao tác append wiki/log.md — nay do code-logger ghi bằng code, không chờ agent.
  • Không bao giờ ghi vào raw/ — đó là inbox của con người, agent chỉ đọc (R1).

Cấu trúc thư mục

  • concepts/ — khái niệm; entities/ — thực thể; sources/ — nguồn + ADR + draft.
  • architecture/, tours/ — kiến trúc & tour. File wiki KHÔNG nằm ở gốc wiki/ (R5).
  • Frontmatter OKF v0.1 (type:…) — máy đọc được (R9).
  • Liên kết chéo bằng [[wikilink]]; trang chỉ tạo SAU khi code đã commit.
04 · Rào chắn

Trụ 2 — Harness (rào chắn tất định)

Harness là phần làm overstack khác mọi "prompt pack": luật là CODE chạy ở hook/CI, chặn được agent kể cả khi nó cố tình lờ. 0 token, không bypass được khi merge.

Hiện có 14 rule (R1–R14), mỗi rule là một validator tất định; vi phạm bị chặn ở write-time (hook), commit (pre-commit), và merge (CI) — ba lớp. Bảng dưới giải thích từng rule. Các cơ chế runtime + tự-gác (auto-index, force-query, orientation, code-index, harness-lint…) là một nhánh “cơ chế” riêng — chia & giải thích từng cái trong mind map (mục Tham chiếu cuối trang).

RuleChặn / đảm bảo điều gì
R1no-write-raw — raw/ là inbox của con người — agent chỉ được đọc, không ghi.
R2origin-required — Mọi wiki content file phải có section '## Origin' để truy được nguồn gốc.
R3index-sync — wiki/index.md phải liệt kê đúng tập file wiki tồn tại; phiên có sửa wiki mà index lệch → chặn kết thúc lượt.
R4audit-log — Mọi tool-call ghi .claude/audit/audit.jsonl (timestamp) + sinh log.md người-đọc-được. Không chặn.
R5folder-structure — File wiki phải nằm trong concepts/entities/sources/draft/architecture/tours — KHÔNG ở wiki/ root.
R6verify-before-commit — Mọi commit phải qua gate: validators + lint + drift-test + R11 seq-glass. Fail = không commit được.
R7proposal-complete — Proposal (có '## Plan' + còn 'proposed') phải khai '## Agent Task Assignment' + link Sequence diagram.
R8session-health — Đầu phiên: báo số rule đang gác + CHECK DRIFT policy/version vs remote (cảnh báo nếu lệch → /sync-template). Không chặn.
R9okf-frontmatter — Wiki content (concepts/entities/sources/draft/architecture/tours — KỂ CẢ nháp) phải có YAML frontmatter + 'type' không rỗng (OKF v0.1). Scope khớp global production: siết frontmatter ngay từ nháp.
R10docs-gate — Mỗi N prompt (default 5) chưa dùng /docs-site-macos hay /orca-workflow → inject directive đề nghị bổ sung docs. Báo cáo, không chặn.
R11seq-html-glass-style — Seq diagram HTML (*-seq.html) PHẢI theo style docs-site-macos liquid-glass — KHÔNG theme flat/tối tự chế (bài học 250626 'sao xấu thế'). Bắt buộc: backdrop-filter glass + nền sáng linear-gradient(180deg,#f7fbff…) + edge-highlight inset.
R12pull-before-change — (B) TRƯỚC khi bắt đầu việc / fan-out đa-agent: orchestrator chạy pull-gate sweep 1 lần (base tươi cho cả đàn agent). (C) TRƯỚC khi push: pre-push hook git-level chặn nếu local SAU remote. KHÔNG có hook chặn từng-edit (per-edit PreToolUse) — cố tình bỏ: cost cao, lệch vendor, đã được (B) phủ.
R13decision-to-adr — Quyết định 'architecture' trong decisions.md phải ref một ADR; ADR cho EDIT tự do; XÓA ADR chỉ khi đã bị đè (Superseded by / supersedes ADR-N).
R14patterns-protected — Kho pattern llmwiki/patterns/ chỉ sửa khi user cho phép (env LLMWIKI_PATTERNS_UNLOCK=1) — agent không tự ý sửa kho tham chiếu (raw/ nhưng thấp hơn một bậc).

Bốn lớp gác (L0–L4)

  • L0 hook — chặn ngay lúc agent định ghi (PreToolUse).
  • L1 settings — wiring policy→agent.
  • L2 pre-commit — chặn lúc commit.
  • L4 CI — chặn lúc merge, nơi không agent nào bypass được.

Harness tự gác chính nó

  • harness-lint — bắt hằng-số-lệch giữa các script.
  • harness-doctor — chạy fixture sai/đúng qua từng validator, chứng minh rào còn cắn.
  • fdk-gate — định-nghĩa-hoàn-thành: 14 bước phải xanh mới cho push.

06 · Quy trình

Workflow — gọi /orca-workflow, nó lo 4 bước

Cách bạn làm việc thực tế: gọi /orca-workflow cho một tính năng — nó tự chạy trọn 4 bước dưới. Các skill dev-loop là bước con do nó điều phối, bạn không gọi tay từng cái.

  1. propose — restate yêu cầu, liệt kê file ảnh hưởng + cái có thể vỡ, vẽ sequence diagram, viết draft. DỪNG chờ bạn duyệt.
  2. gate — validator R7 chặn proposal thiếu (đủ cặp .md + .html một-diagram-mỗi-task).
  3. dispatch — giao việc cho các agent (chọn CLI theo bảng chi phí); chạy song song.
  4. verify — dispatch-verify đối chiếu "khai done" với file thật; /trace-grader chấm cả ĐƯỜNG ĐI.

Hai lệnh điều phối bạn gọi trực tiếp

/orca-workflow (một tính năng) và /orca-onboard (onboard nhiều codebase song song). Các skill dev-loop (propose · impact-check · safe-change · verify-before-commit) là bước CON trong các flow này — gọi tay chỉ khi muốn chạy lẻ một bước.

07 · Điều phối

Orca — điều phối đa-agent

Orca là lớp điều phối: thay vì một agent làm tuần tự, nhiều agent chạy song song theo một flow có kiểm soát.

Khi nào dùng

  • /orca-workflow — vòng propose→gate→dispatch hằng ngày cho một tính năng.
  • /orca-onboard — onboard nhiều codebase song song (phân tích đồng thời).

An toàn

  • orca_guard hook canh hành vi đa-agent.
  • dispatch-verify bắt "khai done nhưng file vắng".
  • trace-grader chấm tool/thứ tự/pass^k — không chỉ kết quả.
08 · Mẫu BNAL

Build-now-adapt-later — thêm tính năng có ẩn số, không liều

Skill /build-now-adapt-later cho BẠN khi thêm vào dự án của bạn một tính năng còn phần chưa chắc (dùng model/API/lib nào, ngưỡng bao nhiêu, dịch vụ ngoài nào). Thay vì chờ chắc-chắn-mới-làm hay liều-đoán-rồi-khoá-mình, nó dựng phần CHẮC CHẮN ngay và nhốt phần CHƯA CHẮC sau MỘT adapter + config verified:false + một ADAPT-CHECKLIST — ship sớm, chỉnh sau ở đúng một chỗ.

Build now

  • Lõi tất định (cái bạn CHẮC) build + test ngay, không phụ thuộc ẩn số.
  • Mọi ẩn số (model nào, ngưỡng bao nhiêu) sống ở ĐÚNG MỘT file config.
  • verified:false = đoán chưa xác nhận; phần gọi LLM/dịch vụ ngoài mặc định enabled:false (fail-safe).
  • adapt-registry gác: ẩn số không rò ra ngoài adapter (Step-7 leak-gate).

Adapt later

  • Khi đã có dữ liệu / đã chốt lựa chọn: lõi pass self-test → verified:true hợp lệ.
  • Phần ngoài (model/API): nối adapter + bật enabled:true sau khi kiểm chứng riêng.
  • Chỉ sửa MỘT file adapter — không phải lục lại khắp codebase.

overstack tự ăn dog food

5 tính năng nâng cao của chính overstack (council · loop-runner · wikieval · trace-grader · failure-flywheel) đều dựng bằng BNAL — nên pattern này đã chứng minh trên thực tế, không phải lý thuyết.

09 · Nâng cao

Eval · Council · Loop — 5 skill cho dự án của bạn

Năm skill nâng cao BẠN gọi khi dev dự án của mình (mỗi cái lõi tất định, phần LLM nhốt sau adapter). Dùng khi cần đánh giá chất lượng, quyết một vấn đề khó, hay chạy một vòng lặp tự-sửa an toàn.

Đánh giá chất lượng

  • /wikieval — bộ eval hồi quy cho wiki dự án bạn (cascade assert tất định + baseline chặn merge khi tụt).
  • /trace-grader — chấm ĐƯỜNG ĐI agent đã đi (tool/thứ tự/pass^k), bắt "đúng kết quả nhưng đi đường tệ".

Quyết định khó & vòng lặp

  • /council — nhiều model trả lời độc lập → chấm chéo ẩn danh → tổng hợp, cho một vấn đề khó (Karpathy 3-stage).
  • /loop-runner — chạy vòng lặp agent CÓ chốt dừng bắt buộc (max-iter/budget/no-progress) — không sợ loop hoang.
  • /failure-flywheel — gom lỗi lặp trong dự án bạn → đề xuất rule/skill mới vào luồng /propose.
10 · Tự-biết

Bản đồ năng lực & Logger (đừng tin agent nhớ)

Hai cơ chế bảo đảm agent biết mình có gì và nhớ đã làm gì — bằng code, không bằng trí nhớ LLM (ADR-005).

Bản đồ năng lực

  • build-capabilities sinh CAPABILITIES.md từ đĩa: mọi skill + rule.
  • Xuống cùng dự án: downstream đọc global skills + rule đã cài → "đồ nghề bạn có Ở DỰ ÁN NÀY".
  • Đủ = skill ＋ rule (harness chặn gì) ＋ state (đã làm gì).

Logger bằng code

  • code-logger để hook ghi log.md bằng CODE mỗi Write/Stop.
  • Không trông chờ agent "nhớ append log" — máy ghi.
  • Xuống cùng dự án (deploy cạnh hooks) → log wiki của chính dự án đó.
11 · Bảo trì

Tự bảo trì overstack trên máy bạn

overstack tự bảo trì được bằng một skill: cập nhật bản mới, trả nợ wiki cũ, refresh bản đồ năng lực, kiểm tra rào chắn — trong một lệnh.

/harness-update      # hoặc: bash harness/scripts/install-harness.sh . --self-heal
  1. Update + self-heal — cài bản mới, tự backfill nợ (Origin/index/OKF) trong một process.
  2. Refresh bản đồ năng lực — build-capabilities --root . để agent thấy đúng đồ nghề sau update.
  3. Health-check — xác nhận rào chắn còn cắn (/health-check).

Đọc kết quả theo exit code: 0 sạch · 3 còn nợ cần xử tay một lần · khác = lỗi hạ tầng (dừng, báo nguyên văn).

12 · Mở rộng

FDK — phát triển chính overstack

👷 Hai tab này (FDK + Dev cái mới) dành cho người PHÁT TRIỂN chính overstack — bỏ qua nếu bạn chỉ DÙNG overstack cho dự án của mình.

Khi bạn muốn sửa CHÍNH overstack (thêm skill/rule/validator/hook), gọi /fdk — front-door on-demand. KHÔNG auto-bơm mọi phiên (phần lớn phiên là dùng overstack cho DỰ ÁN KHÁC — ADR-004).

/fdk lo gì

  • Pre-flight: pull trước khi sửa · biết luật · đừng dẫm module cũ · propose trước.
  • Inventory live (đếm skill/validator/rule từ đĩa, không hardcode).
  • Self-contained: chạy được ở bất kỳ project nào.

Ranh giới (travel)

  • Đồ nghề DEV (fdk-gate, meta-guards) ở LẠI repo overstack.
  • Cái phục vụ dự án (skills, harness, code-logger, capability-map) thì TRAVEL xuống.

Dev framework TỪ một dự án khác → tự mở PR

Đang dở dự án khác mà cần chưng cất một skill vào overstack? Vì fdk-gate/fdk/tools không travel (ADR-004), /fdk dùng fdk-kit.sh: pull kit về sandbox .overstack-kit/ → distill + check (fdk-gate) → submit = push branch + gh pr create tự mở PR vào orca. Một lệnh kéo về, một lệnh đẩy lên.

13 · Checklist

★ Dev một cái mới — cần update gì cho HỢP LỆ

👷 (cho người phát triển overstack) Khi thêm một tính năng/skill/rule mới cho overstack, đây là những chỗ phải đồng bộ để nó hợp lệ (bản máy-đọc = fdk-gate).

  1. Code + test — viết lõi tất định + self-test; ẩn số → nhốt sau MỘT adapter (verified:false).
  2. Rule mới? → thêm vào policy.yaml + validator + wiring (đừng hardcode rời rạc).
  3. Skill mới? → skills/<tên>/SKILL.md (canonical) + sync sang llmwiki/skills/ (mirror) + LOOP_MAP + bảng skill trong AGENT.md ＋ CLAUDE.md (giữ parity) + CAPABILITIES.
  4. Cổng mới? → cắm vào pre-commit + CI + một bước trong fdk-gate.
  5. Wiki "vì sao" — viết concept giải thích lý do tồn tại (như feature-catalog).
  6. Docs người-đọc — cập nhật trang này (regenerate) + README nếu đổi cách dùng.
  7. fdk-gate GREEN — chạy python3 harness/scripts/fdk-gate.py, đủ bước mới push.

Vì sao cần checklist này

overstack lớn → thêm một thứ mà quên đồng bộ một chỗ = drift âm thầm. fdk-gate biến "hợp lệ" thành 14 bước máy-kiểm, không phụ thuộc trí nhớ.
