01 · Database

Chọn Database đúng bài toán

Không có DB nào tốt nhất — chỉ có DB phù hợp nhất với access pattern, scale, và consistency requirement của bạn.

Khi nào dùng SQL?

  • Data có cấu trúc, relationships phức tạp
  • Cần ACID transactions (payment, banking)
  • Joins nhiều bảng là use case chính
  • Schema ổn định, ít thay đổi

Khi nào dùng NoSQL?

  • Cần scale horizontal cực lớn
  • Schema linh hoạt, thay đổi thường xuyên
  • Eventual consistency chấp nhận được
  • Access pattern đơn giản (key lookup)
02 · Sharding

Sharding & Indexing Strategy

Khi single DB không đủ capacity, sharding phân tán dữ liệu ngang. Chọn shard key sai là một trong những quyết định khó sửa nhất.

Indexing Strategy

  • B-tree: range + equality — default cho hầu hết DB
  • Hash index: equality only, O(1) lookup
  • Composite index: leftmost prefix rule
  • Covering index: đủ columns → không hit table
  • Partial index: index trên subset của rows

Shard Key selection

  • Hash-based: phân phối đều, nhưng range query khó
  • Range-based: dễ query range, hotspot risk
  • Geographic: data gần user, nhưng tập trung
  • Tránh shard key là timestamp tuần tự → hotspot
  • Consistent hashing giảm rebalance cost

04 · Distributed

CAP Theorem & Consistency

Một trong những lý thuyết nền tảng nhất của distributed systems. Mọi kỹ sư SD cần hiểu rõ tradeoff này.

Monitoring: 3 Pillars

  • Logs: ghi events có cấu trúc (JSON)
  • Metrics: CPU, memory, latency, error rate
  • Traces: request tracking xuyên suốt services
  • Tools: Prometheus + Grafana, Jaeger, ELK

SLI / SLO / SLA

  • SLI: metric đo được — p99 latency, error rate
  • SLO: target nội bộ — "99.9% requests < 200ms"
  • SLA: cam kết với customer — có penalty vi phạm
  • Error budget = 1 - SLO (dùng để experiment)

05 · Rate Limiting

Rate Limiting

Rate limiting là tuyến phòng thủ đầu tiên chống DDoS, brute-force và API abuse. Không có rate limiting, một client lỗi hoặc malicious có thể exhaust resources của toàn bộ downstream service. Implement tại API Gateway layer để centralize.

AlgorithmBurstMemoryAccuracyDùng khi
Token BucketCó (tích lũy)ThấpCaoAPI rate limit — phổ biến nhất
Leaky BucketKhôngTrung bìnhCaoSmooth downstream processing
Fixed WindowCó (boundary)Rất thấpTrung bìnhSimple, low-risk endpoints
Sliding Window LogKhôngCaoRất caoStrict accuracy cần thiết
Sliding Window CounterNhỏTrung bìnhCaoBalanced accuracy + memory

Algorithm detail

  • Token Bucket: N token, refill R/giây, mỗi request tiêu 1. Burst = tích lũy token. Linh hoạt nhất, dùng nhiều nhất.
  • Leaky Bucket: Queue cố định, xử lý F req/giây. Smooth rate nhưng queue đầy → drop request.
  • Fixed Window: Counter reset mỗi phút. Boundary issue: 100 req cuối + 100 req đầu window mới = 200 req trong 2 giây.
  • Sliding Window Counter: current_count + prev_count × overlap_ratio. Approximate nhưng memory-efficient.

Implementation

  • Redis + Lua script: atomic INCR + EXPIRE → no race condition
  • Key: rate:{user_id}:{window}, TTL = window size
  • Response 429 + Retry-After: <seconds>
  • Headers: X-RateLimit-Limit, X-RateLimit-Remaining, X-RateLimit-Reset
  • Distributed: Redis Cluster share state giữa nhiều API servers
  • Per-user + per-IP + per-API-key — thường kết hợp cả 3
06 · Consensus

Consensus Algorithms

Consensus algorithms giải quyết bài toán: làm thế nào để nhiều node trong distributed system đồng ý về một giá trị, ngay cả khi một số node fail hoặc network bị partition? Đây là nền tảng của mọi distributed database và coordination service.

Raft

Được thiết kế để dễ hiểu hơn Paxos. Chia consensus thành 3 vấn đề độc lập: leader election, log replication, và safety.

  • Leader election: nodes bầu leader qua vote. Leader nhận mọi write, replicate sang followers.
  • Log replication: write → leader log → replicate majority → commit → response client.
  • Term number: tăng mỗi khi có election mới, dùng để detect stale leader.
  • Safety: commit chỉ khi majority (N/2 + 1) xác nhận — không bao giờ mất committed entry.
  • Dùng trong: etcd (Kubernetes), TiKV, CockroachDB, Consul.

Paxos

Thuật toán gốc (Leslie Lamport, 1989). Nền tảng lý thuyết của nhiều distributed systems, nhưng phức tạp để implement đúng.

  • Phase 1 (Prepare): Proposer gửi Prepare(n) → Acceptors promise không accept số nhỏ hơn n.
  • Phase 2 (Accept): Proposer gửi Accept(n, v) → Acceptors accept nếu promise còn valid.
  • Multi-Paxos: optimize với stable leader để skip Phase 1 mỗi round.
  • Dùng trong: Google Chubby, Apache Zookeeper (ZAB — Zookeeper Atomic Broadcast).

Byzantine Fault Tolerance (PBFT)

Tolerates malicious nodes — không chỉ crash mà còn gửi data sai. Cần 3f+1 nodes để tolerate f Byzantine nodes.

  • Phức tạp hơn Raft/Paxos, communication overhead O(n²).
  • Dùng trong: permissioned blockchain, Hyperledger Fabric.
  • Blockchain public (Bitcoin, Ethereum) dùng PoW/PoS thay vì PBFT vì scale tốt hơn.

Khi nào cần hiểu Consensus?

  • Giải thích tại sao Zookeeper/etcd cần quorum (majority) để hoạt động.
  • Tại sao 2 nodes không đủ (split-brain problem) — cần tối thiểu 3.
  • Tại sao distributed DB có write latency cao hơn single node (round-trip consensus).
  • Leader election trong microservices: ai là primary? Circuit breaker state sync.
07 · Observability

Monitoring & Observability

"You can't improve what you can't measure." Monitoring phát hiện incident trước khi user báo cáo. Observability giúp debug tại sao system misbehave — không chỉ biết hỏng mà còn biết TẠI SAO. Setup ngay từ ngày đầu, không phải sau incident đầu tiên.

TermÝ nghĩaVí dụ
SLI (Service Level Indicator)Metric đo được, phản ánh service healthp99 latency < 200ms, error rate < 0.1%
SLO (Service Level Objective)Target nội bộ — team cam kết với nhau99.9% requests trong SLI threshold
SLA (Service Level Agreement)Cam kết với khách hàng — có penalty99.9% uptime/month, vi phạm → credit 10%
Error Budget100% − SLO = phần downtime cho phépSLO 99.9% → 8.7h/year error budget

Logs — Pillar 1

Ghi lại discrete events: user login, order created, error occurred. Structured JSON preferred để dễ query.

  • Format: {"timestamp":"...","level":"ERROR","service":"...","trace_id":"..."}
  • Centralize: ELK Stack (Elasticsearch + Logstash + Kibana), Loki, Datadog Logs
  • Levels: DEBUG → INFO → WARN → ERROR → FATAL
  • Correlation: trace_id xuyên suốt mọi log liên quan đến 1 request

Metrics — Pillar 2

Đo lường numeric values theo thời gian. The Four Golden Signals (Google SRE): Latency, Traffic, Errors, Saturation.

  • Counter: chỉ tăng — total requests, errors
  • Gauge: tăng/giảm — active connections, memory usage
  • Histogram: phân phối — latency buckets (p50/p99/p999)
  • Stack: Prometheus + Grafana (open-source) hoặc Datadog, New Relic (managed)

Traces — Pillar 3

Theo dõi request xuyên suốt nhiều service — distributed tracing. Không thể debug microservices mà không có traces.

  • Trace ID propagate qua headers (W3C Trace Context: traceparent)
  • Span: mỗi service operation. Trace = tree of spans.
  • OpenTelemetry: vendor-neutral standard cho instrumentation
  • Backends: Jaeger, Zipkin (open-source), Datadog APM, AWS X-Ray (managed)

Alerting tốt

  • Alert on symptoms (user impact), not causes — "error rate > 1%" tốt hơn "CPU > 80%"
  • Avoid alert fatigue: chỉ page on-call khi action cần thiết
  • Runbook: mọi alert phải link tới runbook — ai làm gì khi fire
  • P1 (page ngay, prod down) → P2 (page 15 phút) → P3 (ticket, sửa ngày mai)
  • Error budget nearly exhausted → freeze releases, tập trung reliability
