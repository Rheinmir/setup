01 · Microservices

Monolith vs Microservices

Microservices không phải silver bullet — chúng giải quyết vấn đề team scalability và deploy independence, nhưng thêm complexity đáng kể.

Communication Patterns

  • Sync REST/gRPC: cần response ngay
  • Async messaging: fire-and-forget, decouple
  • Event sourcing: audit trail, replay events
  • Saga: distributed transaction — choreography hoặc orchestration

Monolith trước, Microservices sau

  • Giữ monolith: team nhỏ, domain chưa ổn
  • Tách khi: team lớn, deploy freq khác nhau
  • Strangler Fig: tách từng module dần
  • Domain-Driven Design giúp xác định service boundary
02 · Caching

Caching Layers & Strategies

Cache đúng chỗ, đúng chiến lược giảm latency đáng kể và giảm tải DB. Cache sai → stale data, inconsistency.

Cache-Aside (phổ biến nhất)

  • Read: check cache → miss → read DB → write cache → return
  • Write: invalidate cache → write DB
  • Chỉ cache data được đọc thực sự
  • Risk: cache stampede khi nhiều miss đồng thời

Write-Through vs Write-Behind

  • Write-Through: write cache + DB sync — consistency tốt, write chậm hơn
  • Write-Behind: write cache → async flush DB — write nhanh, risk mất data
  • Cache invalidation vẫn là hard problem
  • TTL + version key để handle stale data
03 · High Availability

Circuit Breaker & Bulkhead

Hệ thống resilient không tránh failure — nó fail gracefully. Circuit Breaker và Bulkhead là hai pattern cốt lõi để làm điều đó.

Bulkhead Pattern

  • Tách resource pool cho từng downstream service
  • Service B chậm → không drag cả Service A
  • Thread pool isolation per dependency
  • Impl: Resilience4j, Polly, Hystrix (deprecated)

Retry với Jitter

  • Exponential backoff: 1s → 2s → 4s
  • + Random jitter (0–1s) tránh thundering herd
  • Max retry: 3–5 lần, sau đó fail fast
  • Chỉ retry idempotent operations (GET, PUT)

05 · Communication

Communication Patterns

Microservices phải giao tiếp qua network — lựa chọn pattern ảnh hưởng trực tiếp đến coupling, reliability và performance. Sync communication tạo tight coupling và cascade failure risk; async communication giúp decouple nhưng phức tạp hơn về consistency.

PatternKhi dùngToolsTrade-off
Sync REST/gRPCCần response ngayHTTP, gRPCTight coupling, cascade failure risk
Async MessagingFire-and-forget, decoupleKafka, RabbitMQ, SQSEventual consistency, harder debugging

Saga PatternDistributed transactionChoreography / OrchestrationComplex compensation logic

Saga Pattern (distributed transaction)

Khi microservices cần distributed transaction (Order → Payment → Inventory), Saga chia thành steps, mỗi bước có compensating transaction nếu fail.

  • Choreography: mỗi service publish event → service tiếp theo lắng nghe. Loosely coupled, khó visualize flow.
  • Orchestration: central Saga Orchestrator gửi commands theo thứ tự. Dễ debug, dễ visualize.
  • Compensating transaction: nếu Step 3 fail → trigger undo Step 2 rồi Step 1.
  • Idempotency: mỗi step phải idempotent — retry an toàn khi có network issue.

Event Sourcing

Thay vì lưu current state, lưu tất cả events dẫn đến state đó. Full audit trail và khả năng replay events để rebuild state.

  • Benefits: audit trail đầy đủ, temporal queries, replay để debug hoặc migrate.
  • Trade-offs: storage tăng, querying current state phức tạp hơn (cần projection).
  • Kết hợp với CQRS: event store = write side, projection = read side.
  • Ví dụ: tài khoản ngân hàng — lưu mọi debit/credit thay vì chỉ số dư hiện tại.
06 · Service Discovery

Service Discovery

Trong môi trường microservices dynamic (containers restart, scale in/out, IP thay đổi), hardcoded IP không hoạt động. Service discovery cho phép services tìm nhau dynamically mà không cần cấu hình tĩnh — cần thiết ngay khi deploy 2+ services trong container environment.

Client-side Discovery

Client query service registry để lấy list instances, tự load balance.

  • Client gọi registry (Eureka, Consul) → nhận danh sách instances available
  • Client tự chọn instance (round-robin, random, least-connections)
  • Ưu điểm: flexible load balancing logic tại client
  • Nhược điểm: client phải implement discovery logic, coupled với registry
  • Dùng: Netflix Eureka, Consul, Spring Cloud

Server-side Discovery

Client gọi LB/API Gateway, LB query registry và route tới instance. Client đơn giản hơn — không biết gì về discovery.

  • Client → LB → (LB query registry) → Service instance
  • Ưu điểm: decouple client khỏi discovery logic, client đơn giản
  • Nhược điểm: thêm network hop, LB có thể là bottleneck
  • Dùng: Kubernetes Service (kube-proxy + CoreDNS), AWS ALB + ECS

Kubernetes Service Discovery

  • Mỗi K8s Service có DNS stable: svc.namespace.svc.cluster.local
  • Pod IP thay đổi khi restart, Service IP/DNS không đổi
  • kube-proxy route traffic từ Service IP → Pod IP via iptables/IPVS
  • CoreDNS resolve DNS names trong cluster tự động
  • Headless service: DNS trả thẳng Pod IPs (dùng cho StatefulSet)

Health Checks

  • Liveness: service còn running không? Fail → restart container.
  • Readiness: service sẵn sàng nhận traffic? Fail → remove khỏi LB pool.
  • Startup: service đã xong init chưa? Dùng cho slow-start containers.
  • Registry chỉ route traffic tới healthy instances — health check là cơ chế loại trừ automatic.
07 · Retry & Backoff

Retry với Exponential Backoff & Jitter

Retry sai cách gây thundering herd — toàn bộ clients retry cùng lúc → overwhelm service đang recover. Exponential backoff + jitter phân tán retry theo thời gian, giúp service recover gracefully thay vì bị dồn thêm tải.

Backoff strategies

  • Fixed: wait 1s → 1s → 1s. Đơn giản nhất, tệ nhất cho thundering herd.
  • Linear: wait 1s → 2s → 3s. Tốt hơn Fixed.
  • Exponential: wait 1s → 2s → 4s → 8s → … Tốt, nhưng max chờ có thể quá lâu.
  • Exponential + Jitter (tốt nhất): wait = random(0, min(cap, base × 2^attempt)). Phân tán hoàn toàn, không thundering herd.

Full Jitter formula

  • wait = random(0, min(cap, base × 2^attempt))
  • cap: max wait time (vd: 32s)
  • base: initial wait (vd: 1s)
  • random(0, X): hoàn toàn random → phân tán đều giữa các clients
  • Decorrelated Jitter: wait = random(base, prev_wait × 3) — phân tán tốt hơn với burst nhỏ

Retry best practices

  • Idempotency first: chỉ retry idempotent ops (GET, PUT, DELETE) hoặc POST với idempotency key
  • Max retries: cap tại 3–5 attempts, exponential backoff cap tại 60s
  • Respect Retry-After header: nếu server trả 429 kèm header, dùng giá trị đó
  • DLQ: sau N retries fail → dead letter queue → alert + manual review
  • Circuit breaker + retry: nếu circuit OPEN → không retry (fail fast ngay)
  • Không retry: 4xx errors (400, 401, 403, 404) — lỗi client-side, retry vô nghĩa

Thundering Herd

Khi N services đồng loạt retry sau downtime, service recovering bị flood request ngay khi restart → crash lại.

  • Jitter giải quyết bằng cách phân tán thời điểm retry ngẫu nhiên
  • Circuit breaker ngăn retry trong khi service vẫn down
  • Rate limiting phía server absorb burst dù có jitter
  • Gradual rollout: restart service với traffic thấp trước
08 · Stateless Design

Stateless Services & Horizontal Scale

Stateful services gắn user session vào 1 specific instance — khi instance chết hoặc cần scale, session mất. Stateless services không lưu state trong process memory → bất kỳ instance nào cũng serve được bất kỳ request → horizontal scale tự do. Đây là điều kiện tiên quyết để auto-scale.

Stateless design rules

  • Session data → Redis (TTL, expiry, accessible từ mọi instance)
  • Authentication → JWT (self-contained, stateless) hoặc Redis session
  • File uploads → S3 / object storage (không lưu disk local instance)
  • Background jobs → Message queue (không hardcode worker vào 1 machine)
  • Configuration → Environment variables hoặc config service
  • Cache → Distributed (Redis) thay vì in-process nếu cần share giữa instances

Benefits

  • Auto-scaling: spin up/down bất kỳ lúc nào — không lo session loss
  • Rolling deployment: drain traffic từ old instance, route sang new — zero downtime
  • Fault tolerance: dead instance thay thế tức thì bởi instance mới
  • Load balancing: bất kỳ LB algorithm nào đều hoạt động (không cần sticky)
  • Testing: dễ test hơn vì không có shared mutable state

JWT vs Redis Session

  • JWT: stateless, self-contained (user_id, roles, exp). Không cần DB lookup. Không thể invalidate trước expire (dùng blacklist Redis nếu cần).
  • Redis session: session ID trong cookie, data trong Redis. Có thể invalidate bất kỳ lúc nào. Cần Redis HA. Thêm network roundtrip mỗi request.
  • Hybrid: JWT ngắn hạn (15 phút) + refresh token dài hạn (7 ngày) trong Redis.
09 · Event-Driven

Event-Driven Architecture (EDA)

EDA decouples producers từ consumers hoàn toàn — producer không biết consumer là ai hay có bao nhiêu consumers. Thêm consumer mới (analytics, notification, audit) không cần sửa producer code. Đây là nền tảng để scale và extend system mà không tạo coupling mới.

Architecture

  • Producer → Event Bus (Kafka/SQS) → Consumer Group 1 (billing)
  • → Consumer Group 2 (notifications)
  • → Consumer Group 3 (analytics)
  • → Consumer Group 4 (audit log)
  • Mỗi consumer group xử lý độc lập ở tốc độ của mình (consumer lag)
  • Fan-out: 1 topic → nhiều consumer groups, mỗi group nhận 1 copy message

Fan-out patterns

  • Pub/Sub: 1 topic, nhiều subscriber. Kafka consumer groups mỗi group nhận 1 copy.
  • Queue-based: 1 queue, nhiều consumers competing. Mỗi message chỉ 1 consumer xử lý.
  • Event streaming: Kafka — messages persisted, consumer replay từ đầu nếu cần.
  • SNS → SQS fanout: SNS → nhiều SQS queues. Mỗi downstream có queue riêng → cô lập failure.

Idempotent consumers

Kafka at-least-once delivery → consumer có thể nhận duplicate messages. Mọi consumer phải idempotent:

  • Upsert thay vì insert: INSERT ... ON CONFLICT DO UPDATE
  • Check event_id đã xử lý chưa trong DB/Redis trước khi process
  • Natural idempotency: "set balance = X" an toàn hơn "add X to balance"
  • Deduplication window: Kafka Streams, SQS deduplication ID (FIFO queue)

EDA trade-offs

  • ✓ Loose coupling — thêm consumer không cần sửa producer
  • ✓ Absorb traffic spikes — queue buffer spike, consumer xử lý bình thường
  • ✓ Event replay — debug và migrate data bằng cách replay từ đầu
  • ✗ Eventual consistency — consumer lag, read sau write có thể thấy stale data
  • ✗ Ordering — Kafka partition đảm bảo order trong partition, không cross-partition
  • ✗ Debugging khó — cần distributed tracing để theo dõi event flow
10 · CQRS

CQRS — Command Query Responsibility Segregation

Khi write model và read model có access pattern hoàn toàn khác nhau (write cần ACID, read cần aggregation/denormalization), tách chúng ra giúp optimize từng side độc lập. Write side đảm bảo correctness, read side đảm bảo query performance. Đặc biệt mạnh khi kết hợp với Event Sourcing.

Write Side (Command)

  • Command: intent thay đổi state (CreateOrder, UpdateInventory, ProcessPayment)
  • Command Handler: validate business rules, apply logic, persist to write store
  • Write Model: normalized, ACID, optimized for correctness — không phải performance
  • Emit domain events sau khi write → async update read models
  • Write DB: thường PostgreSQL, MySQL với strict schema và transactions

Read Side (Query)

  • Query: không thay đổi state (GetOrderSummary, GetUserDashboard)
  • Read Model: denormalized, pre-aggregated, optimized cho specific query patterns
  • Materialized view / projection: rebuild từ domain events, updated async
  • Multiple read models: 1 write → nhiều read models cho mobile, admin, analytics
  • Read DB: thường Redis, Elasticsearch, hoặc denormalized relational DB

Architecture flow

  • Client → Command → Command Handler → Write DB + Event Store
  • Event Store → Event Processor → Read Model (async)
  • Client → Query → Query Handler → Read Model (fast)
  • Eventual consistency: read model lag behind write (milliseconds to seconds)
  • Kết hợp với Event Sourcing: events = source of truth cho cả write và read

Trade-offs

  • ✓ Optimize read/write independently — scale read replicas không ảnh hưởng write
  • ✓ Separation of concerns — write logic và query logic không pha trộn
  • ✓ Multiple read models — mỗi use case có projection riêng, query cực nhanh
  • ✗ Eventual consistency — read model lag sau write
  • ✗ Complexity — 2 models, sync logic, projection rebuild khi schema thay đổi
  • ✗ NOT for simple CRUD — overhead không đáng cho app nhỏ hoặc early stage
