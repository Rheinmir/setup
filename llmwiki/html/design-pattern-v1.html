System Design Scalability Architecture Fundamentals
01 · Foundation

Tại sao cần System Design?

Khi product lớn lên, những câu hỏi kỹ thuật trở nên phức tạp hơn nhiều so với "code đúng". System Design là framework để trả lời chúng.

Câu hỏi cần trả lời

  • Chịu được bao nhiêu user đồng thời?
  • Dữ liệu lưu ở đâu, truy xuất thế nào?
  • Khi 1 component fail thì toàn hệ thống sập không?
  • Scale như thế nào khi traffic tăng 10×?
  • Latency target là bao nhiêu (p99)?

Framework tiếp cận (5 bước)

  • 1. Clarify requirements (functional + non-functional)
  • 2. Estimate scale (DAU, RPS, storage bytes)
  • 3. High-level design (components + data flow)
  • 4. Deep dive (DB schema, API, caching)
  • 5. Bottlenecks + tradeoffs
02 · Core Axes

Các trục đánh giá hệ thống

Mọi quyết định System Design đều xoay quanh 5 trục này. Cải thiện một trục thường kéo theo đánh đổi với trục khác.

Availability 9s

  • 99% = 3.65 ngày/năm downtime
  • 99.9% = 8.7 giờ/năm
  • 99.99% = 52 phút/năm
  • 99.999% = 5.2 phút/năm

Performance metrics quan trọng

  • p50: median — trải nghiệm trung bình
  • p99: 99th percentile — worst 1%
  • p999: 99.9th — tail latency
  • Throughput: requests/second (RPS)
  • Latency budget: end-to-end time
03 · Building Blocks

Các thành phần cơ bản

Hầu hết hệ thống lớn đều được lắp ráp từ cùng một bộ building blocks. Hiểu rõ từng block và khi nào dùng là nền tảng của SD interview.

Load Balancer

  • Phân phối traffic đến nhiều server
  • L4 (TCP) — routing theo IP/port
  • L7 (HTTP) — routing theo URL/header
  • Algo: round-robin, least conn, IP-hash

Cache (Redis/Memcached)

  • Giảm DB load, tăng read throughput
  • Cache-aside: check cache → miss → DB → write
  • Write-through: write cache + DB đồng thời
  • TTL + eviction: LRU, LFU, FIFO

Database — SQL vs NoSQL

  • SQL: ACID, joins, strong consistency
  • NoSQL: horizontal scale, flexible schema
  • Read replicas: phân tán read traffic
  • Sharding: partition dữ liệu ngang

Message Queue

  • Decouple producer và consumer
  • Async processing, retry logic
  • Dead letter queue cho failed messages
  • Fan-out: 1 event → nhiều consumer
04 · Tradeoffs

Tradeoffs thường gặp

Không có kiến trúc "đúng tuyệt đối" — chỉ có tradeoff phù hợp với context. Mục tiêu là nhận ra và quyết định có chủ đích.

Tình huốngOption AOption BChọn A khiChọn B khi
Consistency vs AvailabilityStrong consistencyHigh availabilityBanking, paymentSocial feed, timeline
SQL vs NoSQLRelational DBDocument / Key-valueStructured + joinsScale + flexible schema
Sync vs AsyncSynchronous APIMessage queueReal-time response cầnDecouple + scale
Monolith vs MicroservicesSingle deployableMany servicesEarly stage, small teamScale teams, diff deploy freq
Push vs PullServer push (WebSocket)Client pollReal-time notificationsBatch / low-freq updates

Estimation tips

  • DAU × actions/day ÷ 86400 = QPS
  • Peak = avg QPS × 2–3×
  • Storage: 1 user record ≈ 1KB → 1M users = 1GB
  • Cache hit rate target: ≥ 80–90%

Checklist interview

  • Clarify requirements (5 phút)
  • Estimate scale: DAU, QPS, storage
  • High-level: LB → App → DB + Cache
  • API design + DB schema
  • Deep dive 1 bottleneck + tradeoffs
