System Design Scalability Architecture Fundamentals
01 · Foundation

Tại sao cần System Design?

Why: Khi product tăng từ 100 lên 10 triệu user, câu hỏi không còn là "code đúng không?" mà là "hệ thống có sập không?". System Design là blueprint để trả lời những câu hỏi đó TRƯỚC khi xảy ra sự cố thật.

When: Bắt buộc khi team scale, khi traffic có peak bất thường, khi cần SLA rõ ràng với khách hàng.

Câu hỏi cần trả lời

  • Chịu bao nhiêu user đồng thời?
  • Dữ liệu lưu ở đâu, truy xuất thế nào?
  • Khi 1 component fail thì toàn hệ thống sập không?
  • Scale thế nào khi traffic tăng 10×?
  • Latency target là bao nhiêu (p99)?

Bộ khung 5 bước tiếp cận

  • 1. Clarify requirements — Tách functional (user phải làm được gì) vs non-functional (latency, availability, scale). Đừng design trước khi hiểu rõ constraint.
  • 2. Estimate scale — DAU × actions/day ÷ 86400 = QPS trung bình. Peak = avg × 2–3×. Storage: 1 record ≈ 1KB → 1M users = 1GB.
  • 3. High-level design — Vẽ components và data flow: Client → LB → App → DB/Cache. "Big picture" để align với interviewer trước khi deep dive.
  • 4. Deep dive — Chọn 1–2 bottleneck để zoom vào: DB schema, API design, sharding strategy, cache invalidation.
  • 5. Bottlenecks + tradeoffs — Mọi quyết định đều có trade-off. Nhận ra và giải thích rõ: "Tôi chọn X vì Y, đánh đổi Z."
02 · Core Axes

Các trục đánh giá hệ thống

Why: Mọi quyết định kiến trúc đều cần đặt lên 5 trục này. Cải thiện một trục thường kéo theo đánh đổi với trục khác — biết trục nào quan trọng nhất giúp quyết định nhanh và đúng.

TrụcCâu hỏi cốt lõiMetric đo
ScalabilityHorizontal vs vertical scalingThroughput (RPS) khi thêm node
ReliabilitySingle point of failure, redundancyMTBF, MTTR
AvailabilityUptime SLA99.9% = 8.7h downtime/year
PerformanceLatency vs throughputp50, p99, p999 latency
ConsistencyCAP theoremStrong vs Eventual consistency

Availability "nines"

  • 99% = 3.65 ngày/năm downtime
  • 99.9% = 8.7 giờ/năm
  • 99.99% = 52 phút/năm
  • 99.999% = 5.2 phút/năm

Performance metrics

  • p50: median — trải nghiệm người dùng trung bình
  • p99: 99th percentile — worst 1% user experience
  • p999: tail latency — quan trọng với real-time systems
  • Throughput: requests/second (RPS) — khả năng chịu tải
03 · Building Blocks

Building Blocks

Why: Hầu hết hệ thống lớn đều lắp từ cùng một bộ building blocks. Hiểu rõ từng block — khi nào dùng, khi nào không — là nền tảng thiết yếu cho mọi System Design interview.

Load Balancer

Why: Tránh single point of failure ở tầng server. Khi 1 server chết, LB route traffic sang server khác tự động, user không thấy downtime.

When: Ngay khi cần hơn 1 app server.

  • L4 (TCP): routing theo IP/port, nhanh hơn, ít overhead
  • L7 (HTTP): routing theo URL/header/cookie, linh hoạt, support path-based routing
  • Algorithms: round-robin (đơn giản), least-connections (phân phối đều khi latency khác nhau), IP-hash (sticky session cùng user)
  • Health check: LB tự loại server unhealthy khỏi pool

Database

Why: Chọn đúng DB là quyết định khó thay đổi nhất sau này. SQL mạnh về consistency, NoSQL mạnh về scale.

When: SQL khi cần ACID transactions + joins phức tạp (banking, e-commerce). NoSQL khi cần flexible schema + horizontal scale (social feed, user profiles, logs).

  • SQL (PostgreSQL, MySQL): ACID, joins, strong consistency, max ~10M QPS với replicas
  • NoSQL (MongoDB, DynamoDB, Cassandra): BASE, horizontal scale, eventual consistency
  • Read replicas: replica nhận read traffic → giảm tải primary 70–80%
  • Sharding: partition data theo key (user_id hash, geographic) khi single DB không đủ

Cache

Why: DB disk I/O là bottleneck phổ biến nhất. Cache đưa hot data vào RAM — latency giảm từ 10ms xuống dưới 1ms, giảm 80% DB load.

When: Read-heavy workload, data ít thay đổi, chấp nhận stale data trong vài giây/phút.

  • Cache-aside (lazy loading): check cache → miss? → read DB → write cache → return. Pattern phổ biến nhất.
  • Write-through: write cache + DB đồng thời. Consistency cao hơn nhưng write latency tăng.
  • Write-behind (write-back): write cache → async flush DB. Write nhanh nhưng risk mất data nếu cache crash.
  • TTL + eviction: LRU (least recently used) phổ biến, LFU (frequency-based) cho hot-cold workloads.
  • Cache invalidation là bài toán khó nhất: thay đổi data phải invalidate cache đúng lúc.

Message Queue

Why: Decouple producer và consumer — producer không cần biết consumer có sẵn không. Giúp absorb traffic spike mà không mất request.

When: Async processing (email, notifications, image resize), decouple microservices, fan-out events.

  • Kafka: high-throughput (millions msg/s), persistent log, replay events, partitioned theo key
  • RabbitMQ: flexible routing (exchange/binding), AMQP protocol, lower throughput hơn Kafka
  • Dead letter queue: chứa messages failed sau N retries → alert và debug sau
  • Fan-out: 1 event → nhiều consumer groups xử lý độc lập (billing, notification, analytics)

CDN

Why: Static assets không cần request về origin server mỗi lần — CDN cache tại edge node gần user nhất, latency giảm từ 200ms xuống 10ms.

When: Khi user phân tán địa lý, khi có nhiều static content (images, JS, CSS, video), khi muốn giảm bandwidth cost origin.

  • Edge caching: CloudFront, Akamai, Fastly — hàng trăm PoP toàn cầu
  • Cache-Control headers: max-age (browser cache), s-maxage (CDN cache), must-revalidate
  • Invalidation: purge cache khi deploy mới — chú ý cost với Akamai/CloudFront
  • Dynamic content: một số CDN support acceleration (không cache nhưng route tối ưu qua backbone)
04 · Rate Limiting

Rate Limiting

Why: Không có rate limiting, 1 bad actor có thể gây DoS toàn hệ thống chỉ bằng request burst. Rate limiting bảo vệ downstream resources và đảm bảo fairness giữa các client. Authentication endpoints đặc biệt cần thiết để chống brute-force.

When: Mọi public API. Đặc biệt quan trọng với /login, /payment, /send-email endpoints. Implement ở API Gateway layer để centralize.

AlgorithmBurst allowedMemoryAccuracyDùng khi
Token BucketCó (tích lũy)ThấpCaoAPI rate limit phổ biến nhất
Leaky BucketKhôngTrung bìnhCaoSmooth processing rate
Fixed WindowCó (ở boundary)Rất thấpTrung bìnhSimple use case
Sliding WindowKhôngCaoRất caoChính xác cao cần thiết

Algorithms detail

  • Token Bucket: mỗi user có bucket N token, mỗi request tiêu 1 token, token refill đều đặn. Cho phép short burst (dùng hết token tích lũy). Linh hoạt nhất.
  • Leaky Bucket: request vào queue, xử lý với rate cố định (nước chảy qua lỗ). Smooth output nhưng queue có thể đầy → drop request.
  • Fixed Window: đếm request trong window cố định (100 req/phút). Đơn giản nhưng burst xảy ra ở boundary: 100 req cuối window + 100 req đầu window mới = 200 req trong 1 giây.
  • Sliding Window Log: ghi timestamp từng request, đếm trong cửa sổ di chuyển. Chính xác tuyệt đối nhưng tốn memory nhiều nhất.

Implementation

  • Redis + Lua script để atomic check-and-decrement (tránh race condition)
  • Key pattern: rate_limit:{user_id}:{window} với TTL = window size
  • Response 429 Too Many Requests + Retry-After header (giây còn lại)
  • Distributed rate limiting: Redis cluster để share state giữa nhiều API server
05 · Tradeoffs

Tradeoffs thường gặp

Why: Không có kiến trúc "đúng tuyệt đối" — chỉ có tradeoff phù hợp với context. Người giỏi System Design là người nhận ra tradeoff và giải thích có chủ đích thay vì chọn mù.

Tình huốngOption AOption BChọn A khiChọn B khi
Consistency vs AvailabilityStrong consistencyHigh availabilityBanking, payment, inventorySocial feed, timeline, likes
SQL vs NoSQLRelational DBDocument/Key-valueStructured + joins, ACID cầnScale + flexible schema
Sync vs AsyncSynchronous APIMessage queueReal-time response cần ngayDecouple + absorb spike
Monolith vs MicroservicesSingle deployableMany servicesEarly stage, small teamScale teams, diff deploy freq
Push vs PullServer push (WebSocket)Client pollReal-time notificationsBatch/low-freq updates
Cache-aside vs Write-throughCache-asideWrite-throughRead-heavy, stale data OKWrite-heavy, consistency cần

Estimation cheat sheet

  • DAU × actions/day ÷ 86400 = QPS trung bình
  • Peak QPS = avg × 2–3× (hoặc × 10 nếu có event đặc biệt)
  • 1 user record ≈ 1KB → 1M users = 1GB storage
  • Cache hit rate target: ≥ 80–90% để có hiệu quả

Interview checklist

  • Clarify requirements (5 phút)
  • Estimate scale: DAU, QPS, storage (5 phút)
  • High-level: LB → App → DB + Cache
  • API endpoints + DB schema
  • Deep dive 1 bottleneck + tradeoffs
  • Monitoring: metrics, alerts
